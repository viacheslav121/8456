# Docker: История, Описание и Экосистема

## 1. Определение

**Docker** — это платформа с открытым исходным кодом, предназначенная для разработки, доставки и запуска приложений в изолированных средах, называемых **контейнерами**.

В отличие от традиционных виртуальных машин (VM), которые эмулируют целое оборудование и запускают полноценную операционную систему, контейнеры используют ядро хостовой ОС. Это делает их:
*   **Легковесными:** Занимают меньше места и потребляют меньше ресурсов.
*   **Быстрыми:** Запускаются за секунды (иногда миллисекунды).
*   **Портативными:** Работают одинаково на любой системе, где установлен Docker (Linux, Windows, macOS).

Основная идея: *"Build once, run anywhere"* (Собери один раз, запусти где угодно).

## 2. История появления и развития

### Эра до Docker (Pre-Docker)
Технологии изоляции существовали задолго до 2013 года:
*   **1979:** Системный вызов `chroot` в Unix (изоляция файловой системы).
*   **2000:** FreeBSD Jails (более продвинутая изоляция).
*   **2001:** Linux VServer.
*   **2004:** Solaris Zones.
*   **2008:** **LXC (Linux Containers)** — ключевая технология, ставшая фундаментом для раннего Docker. Она позволяла запускать изолированные пространства процессов в одном ядре Linux.

Однако работа с LXC была сложной и требовала глубоких знаний Linux. Не хватало удобного инструмента для управления жизненным циклом приложений.

### Рождение Docker (2013)
*   **Компания:** Проект начался внутри стартапа **dotCloud** (платформа PaaS).
*   **Создатель:** Соломон Хайкс (Solomon Hykes).
*   **Презентация:** Впервые Docker был представлен на конференции **PyCon в марте 2013 года**.
*   **Инновация:** Docker добавил удобный CLI (интерфейс командной строки), упростил работу с LXC и ввел концепцию **образов (images)** как неизменяемых пакетов для доставки кода.

### Ключевые вехи развития
*   **2013 (Июль):** Docker становится проектом с открытым исходным кодом.
*   **2014:** Выпуск Docker 1.0. Компания dotCloud переименовывается в **Docker Inc.**.
*   **2015:** Появление **Docker Swarm** (встроенный оркестратор) и рост популярности **Kubernetes**.
*   **2017:** Docker Inc. передает нижнеуровневую технологию (runtime, формат образов) сообществу под названием **Moby Project**. Docker Inc. фокусируется на коммерческих продуктах (Docker Desktop, Enterprise).
*   **2019:** Разделение Docker Engine на Community Edition (CE) и Enterprise Edition (EE).
*   **2020:** Продажа корпоративного бизнеса Docker компании Mirantis.
*   **2021:** Изменение лицензионной политики **Docker Desktop**. Для крупных компаний (более 250 сотрудников или $10 млн выручки) использование стало платным.

## 3. Современное состояние

На сегодняшний день Docker является де-факто стандартом в индустрии разработки и DevOps.

### Экосистема
*   **Контейнеризация:** Почти все современные микросервисы упаковываются в контейнеры.
*   **Оркестрация:** Хотя Docker Swarm существует, индустриальным стандартом для управления кластерами контейнеров стал **Kubernetes (K8s)**. Docker часто используется для локальной разработки, а Kubernetes — для продакшена.
*   **CI/CD:** Docker интегрирован во все популярные системы непрерывной интеграции (GitLab CI, GitHub Actions, Jenkins).

### Конкуренты и альтернативы
*   **Podman:** Инструмент от Red Hat, работающий без демона (daemonless), позиционируется как более безопасная альтернатива.
*   **Containerd:** Низкоуровневый рантайм, используемый внутри Docker и Kubernetes.
*   **LXC/LXD:** До сих пор используются для контейнеризации целых систем (системные контейнеры), а не приложений.

## 4. Docker Hub и готовые образы

**Docker Hub** — это облачный реестр (registry) для хранения и распространения Docker-образов. Аналог GitHub для контейнеров.

### Типы образов
1.  **Official Images:** Поддерживаются самой компанией Docker и авторами ПО (например, `nginx`, `python`, `postgres`). Проходят проверку на безопасность и качество.
2.  **Publisher Images:** Создаются и поддерживаются сторонними компаниями или разработчиками.
3.  **User Images:** Создаются обычными пользователями (требуют осторожности при использовании в продакшене).

### Важность безопасности
Использование готовых образов ускоряет разработку, но несет риски:
*   Уязвимости в базовом слое.
*   Внедрение вредоносного кода (Supply Chain Attacks).
*   **Рекомендация:** Использовать официальные образы, сканировать их на уязвимости и минимизировать количество слоев.

## 5. Dockerfile

**Dockerfile** — это текстовый файл с инструкциями, по которым собирается Docker-образ. Это "рецепт" вашего контейнера.

### Основные инструкции
*   `FROM`: Базовый образ (например, `ubuntu`, `alpine`).
*   `WORKDIR`: Установка рабочей директории внутри контейнера.
*   `COPY` / `ADD`: Копирование файлов из хоста в образ.
*   `RUN`: Выполнение команд во время сборки (установка пакетов, компиляция).
*   `ENV`: Установка переменных окружения.
*   `EXPOSE`: Информирование о портах, которые будет слушать контейнер.
*   `CMD` / `ENTRYPOINT`: Команда, которая запускается при старте контейнера.

### Пример Dockerfile (Python приложение)
```dockerfile
# Базовый образ
FROM python:3.9-slim

# Установка рабочей директории
WORKDIR /app

# Копирование зависимостей и установка
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Копирование исходного кода
COPY . .

# Переменная окружения
ENV FLASK_APP=main.py

# Порт
EXPOSE 5000

# Команда запуска
CMD ["flask", "run", "--host=0.0.0.0"]
```

## 6. Docker Compose
Docker Compose — это инструмент для определения и запуска многоконтейнерных приложений Docker.
### Зачем нужен?
Если приложение состоит из нескольких сервисов (например: Веб-сервер + База данных + Кэш), запускать их отдельными командами docker run неудобно. Compose позволяет описать всю инфраструктуру в одном файле docker-compose.yml и управлять ею одной командой.
### Основные возможности
* Описание сервисов, сетей и томов (volumes).
* Запуск и остановка всего стека одной командой.
* Изоляция среды разработки от продакшена.

### Пример docker-compose.yml:
```shell
version: '3.8'

services:
  web:
    build: .
    ports:
      - "5000:5000"
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:13
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```
### Команды управления:
* docker compose up — Запустить сервисы.
* docker compose down — Остановить и удалить ресурсы.
* docker compose ps — Показать статус контейнеров.
* docker compose logs — Просмотр логов.
